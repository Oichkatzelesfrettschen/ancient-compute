# Blender Render Pipeline Specification

## WHY / WHAT / HOW

**WHY**: The machine gallery frontend displays 30 historical machines. Without rendered
images, each gallery card shows only text. A Blender pipeline produces consistent,
historically-grounded hero images -- matching the material data already in the registry
(brass, cast iron, ivory, boxwood, etc.) -- and enables the gallery to function as a
visual museum rather than a machine list.

**WHAT**: This spec defines the full pipeline: directory layout, scene template, material
mapping from MaterialLibrary to Blender Principled BSDF, batch-render automation
(`tools/render_gallery.py`), output formats, and frontend integration. It covers all
30 machines in `machine_registry.py`.

**HOW**: One Blender `.blend` file per machine, organized under `assets/renders/blender/`.
A Python automation script drives headless batch rendering via `blender --background
--python`. Outputs land in `frontend/static/machines/` as 800x600 WebP (gallery cards)
and 1600x1200 WebP (detail pages). A placeholder SVG is served for any machine without
a `.blend` file.

Machine directories use the exact `id` string from `MachineEntry` (URL-safe slugs, e.g.
`analytical-engine`, `zuse-z3`, `hollerith-tabulator`). This makes the mapping from
registry to file system deterministic without a lookup table.

## Render Manifest Format

Each `<machine-id>.render.json` file describes the scene for the automation script:

```json
{
  "machine_id": "analytical-engine",
  "blend_file": "analytical-engine.blend",
  "camera": "Camera.001",
  "frame": 1,
  "engine": "CYCLES",
  "samples": 512,
  "denoiser": "OPENIMAGEDENOISE",
  "outputs": [
    { "label": "card",   "width": 800,  "height": 600,  "format": "WEBP", "quality": 90 },
    { "label": "detail", "width": 1600, "height": 1200, "format": "WEBP", "quality": 90 },
    { "label": "thumb",  "width": 200,  "height": 150,  "format": "WEBP", "quality": 80 }
  ],
  "env_map": "../../_template/env/studio_neutral.exr",
  "overrides": {}
}
```

**Fields**:
- `camera`: Blender object name of the active camera (allows per-machine framing).
- `frame`: Which animation frame to render (1 for static; >1 for articulated poses).
- `engine`: `CYCLES` for production; `BLENDER_EEVEE_NEXT` for fast previews.
- `samples`: Cycles sample count. 512 is sufficient for 800x600 + denoising.
- `denoiser`: `OPENIMAGEDENOISE` requires the CPU add-on; use `NLM` as fallback.
- `env_map`: HDR environment map for consistent lighting across all machines.
  `studio_neutral.exr` is a grey studio dome (0.8 grey, no colored specular).
- `overrides`: Optional per-machine Blender property overrides (e.g. `{"world.color": [0.05, 0.05, 0.05]}`).

## Scene Template: `_template/scene_template.blend`

All machine scenes are built by appending the template's collection `Lighting.Rig` before
setting the camera. The template defines:

### Camera Rig

```
Camera.001  (perspective, FOV 50 deg)
  Location: (2.5, -3.0, 1.8) m
  Rotation: (63 deg, 0, 47 deg) Euler XYZ
  Lens: 85 mm equivalent at sensor 36 mm
  Depth of field: f/8, focus distance 3.5 m
```

This is the "3/4 front left above" angle used consistently across all machines. Individual
scene files may override with a second camera `Camera.002` for a detail view.

### Lighting Rig

Three-point lighting derived from a neutral HDR dome plus fill:

| Light      | Type  | Power   | Color (K)  | Location        | Purpose          |
|------------|-------|---------|------------|-----------------|------------------|
| Key        | Area  | 800 W   | 5500 K     | (3, -2, 4) m    | Main light       |
| Fill       | Area  | 200 W   | 6500 K     | (-4, -1, 2) m   | Shadow fill      |
| Rim        | Spot  | 400 W   | 4200 W     | (-1, 3, 3) m    | Edge separation  |
| Ground     | Area  | 50 W    | 3200 K     | (0, 0, -0.5) m  | Bounce warm fill |

The HDR dome (`studio_neutral.exr`) is loaded at strength 0.3 to provide ambient
occlusion without colored cast.

### Ground Plane

A single plane mesh `Ground.001`, 3 x 3 m, with a grey diffuse material (roughness 0.9,
albedo 0.2). Provides shadow catch. Rendered with `Ray Visibility > Shadow` only so it
does not appear in the final image except as a shadow receiver.

## Material Library: MaterialLibrary -> Blender Principled BSDF

Each material `.blend` file contains one Principled BSDF node group named after the
MaterialLibrary key. The automation script links these at render time.

### Mapping Table

All parameters are for the Blender 4.x Principled BSDF shader.

| Library Key           | Base Color (sRGB)       | Metallic | Roughness | Specular | IOR   | Notes                             |
|-----------------------|-------------------------|----------|-----------|----------|-------|-----------------------------------|
| `cast_iron`           | (0.13, 0.12, 0.11)      | 1.0      | 0.80      | 0.5      | 2.95  | Grey, very rough; no shine        |
| `brass`               | (0.88, 0.72, 0.29)      | 1.0      | 0.20      | 0.5      | 1.0   | Warm gold; semi-polished          |
| `steel`               | (0.80, 0.80, 0.80)      | 1.0      | 0.35      | 0.5      | 2.76  | Cold grey; brushed finish         |
| `spring_steel`        | (0.75, 0.75, 0.75)      | 1.0      | 0.25      | 0.5      | 2.76  | Slightly shinier than plain steel |
| `phosphor_bronze`     | (0.80, 0.65, 0.35)      | 1.0      | 0.15      | 0.5      | 1.0   | Deep gold; high contact polish    |
| `wrought_iron`        | (0.18, 0.17, 0.16)      | 1.0      | 0.70      | 0.5      | 2.95  | Darker than cast iron; fibrous    |
| `gun_metal`           | (0.55, 0.45, 0.30)      | 1.0      | 0.25      | 0.5      | 1.0   | Brown-bronze; bearing finish      |
| `corinthian_bronze`   | (0.72, 0.52, 0.22)      | 1.0      | 0.30      | 0.5      | 1.0   | Ancient patinated bronze          |
| `ivory`               | (0.96, 0.92, 0.80)      | 0.0      | 0.50      | 0.4      | 1.54  | Subsurface scatter depth 5 mm     |
| `boxwood`             | (0.82, 0.72, 0.45)      | 0.0      | 0.60      | 0.3      | 1.50  | Fine-grain wood; pale yellow      |
| `aluminum_alloy_1940s`| (0.90, 0.90, 0.90)      | 1.0      | 0.45      | 0.5      | 1.0   | Anodized; matte silver            |
| `bakelite`            | (0.20, 0.12, 0.08)      | 0.0      | 0.45      | 0.3      | 1.55  | Dark brown; smooth moulded        |
| `organic_fiber`       | (0.75, 0.65, 0.45)      | 0.0      | 0.90      | 0.1      | 1.45  | Cord/rope appearance              |
| `mercury`             | (0.85, 0.90, 0.95)      | 1.0      | 0.05      | 0.5      | 1.0   | Mirror liquid; used for delay tubes|

**Ivory subsurface scatter**: Use Principled BSDF with Subsurface method `RANDOM_WALK`,
subsurface weight 0.25, subsurface radius (1.0, 0.8, 0.6) to simulate translucency.

**Boxwood texture**: Apply a Musgrave texture (scale 200, detail 6) to the roughness
channel (range 0.45 to 0.75) to simulate wood grain variation.

**Patina layers** (optional): For antikythera and astrolabe (ancient brass/bronze),
add a secondary layer using Blender's `MixShader`: 70% clean metal + 30% green-grey
diffuse (0.4, 0.55, 0.3) to simulate verdigris. Activate by setting `patina=true`
in the render manifest `overrides`.

## Batch Render Automation: `tools/render_gallery.py`

```
Usage:
  python3 tools/render_gallery.py [--machine MACHINE_ID] [--all] [--engine CYCLES|EEVEE]
                                   [--quality draft|final] [--output-dir PATH]

Options:
  --machine MACHINE_ID   Render a single machine (default: all that have .blend files)
  --all                  Render all 30 machines (skips any without a .blend file)
  --engine               Override render engine (default: from manifest)
  --quality draft        128 samples + EEVEE for quick iteration
  --quality final        512 samples + Cycles + denoising (default for CI)
  --output-dir PATH      Output directory (default: assets/renders/outputs/)
  --list                 Print manifest summary for all machines; exit
  --missing              Print machine IDs that have no .blend file; exit
```

**Implementation outline** (`tools/render_gallery.py`):

```python
"""Headless batch renderer for the machine gallery.

Drives Blender via subprocess using --background --python to render each
machine in the registry that has a .blend scene file.

Requires: Blender 4.1+ on PATH as `blender`.
"""

import json
import subprocess
import sys
from pathlib import Path

REPO_ROOT = Path(__file__).resolve().parents[1]
BLENDER_DIR = REPO_ROOT / "assets" / "renders" / "blender" / "machines"
OUTPUT_DIR = REPO_ROOT / "assets" / "renders" / "outputs"
FRONTEND_STATIC = REPO_ROOT / "frontend" / "static" / "machines"

BLENDER_RENDER_SCRIPT = REPO_ROOT / "tools" / "_blender_render_worker.py"


def load_machine_ids() -> list[str]:
    """Return all machine IDs from machine_registry via JSON import."""
    sys.path.insert(0, str(REPO_ROOT / "backend"))
    from backend.src.emulator.machine_registry import get_registry
    return [entry.id for entry in get_registry()]


def find_manifest(machine_id: str) -> Path | None:
    manifest = BLENDER_DIR / machine_id / f"{machine_id}.render.json"
    return manifest if manifest.exists() else None


def render_machine(machine_id: str, quality: str = "final") -> bool:
    manifest_path = find_manifest(machine_id)
    if manifest_path is None:
        print(f"[skip] {machine_id}: no render manifest")
        return False
    cmd = [
        "blender",
        "--background",
        str(manifest_path.parent / f"{machine_id}.blend"),
        "--python", str(BLENDER_RENDER_SCRIPT),
        "--",
        "--manifest", str(manifest_path),
        "--quality", quality,
        "--output-dir", str(OUTPUT_DIR),
    ]
    result = subprocess.run(cmd, capture_output=True, text=True)
    if result.returncode != 0:
        print(f"[FAIL] {machine_id}:\n{result.stderr[-1000:]}")
        return False
    print(f"[ok]   {machine_id}")
    return True
```

`_blender_render_worker.py` is executed inside Blender's Python environment. It reads
the manifest, sets up output nodes, configures the denoiser, and calls
`bpy.ops.render.render(write_still=True)` for each output size.

**Copy outputs to frontend**:

After rendering, `render_gallery.py` copies (or symlinks) outputs to
`frontend/static/machines/`:
- `<id>.webp` (800x600 card size)
- `<id>@2x.webp` (1600x1200 detail size)
- `<id>_thumb.webp` (200x150 thumbnail)

Machines without a render fall back to `_placeholder.svg` (see below).

## Placeholder SVG

`frontend/static/machines/_placeholder.svg` is a 800x600 SVG that renders a silhouette
of mechanical gears with the machine name centred below in the gallery font. The frontend
uses it when `/<machine-id>.webp` returns 404.

Implementation in SvelteKit:

```svelte
<!-- frontend/src/lib/components/MachineImage.svelte -->
<script lang="ts">
  export let machineId: string;
  export let alt: string;
  let src = `/machines/${machineId}.webp`;
  let fallback = false;
</script>

<img
  {src}
  {alt}
  on:error={() => { fallback = true; src = "/machines/_placeholder.svg"; }}
  class:placeholder={fallback}
/>
```

## Frontend Integration

### Static Asset Serving

SvelteKit serves `frontend/static/` at the root. The machine gallery at
`/machines` and the detail page at `/machines/[id]` consume:

| Path                              | Size       | Used in                       |
|-----------------------------------|------------|-------------------------------|
| `/machines/<id>.webp`             | 800x600    | Gallery card (`<img>`)        |
| `/machines/<id>@2x.webp`          | 1600x1200  | Detail page hero              |
| `/machines/<id>_thumb.webp`       | 200x150    | Mobile list / search results  |
| `/machines/_placeholder.svg`      | 800x600    | Error fallback                |

### Responsive Images

In the gallery card:

```svelte
<picture>
  <source media="(min-width: 768px)" srcset="/machines/{id}@2x.webp" type="image/webp" />
  <img src="/machines/{id}.webp" alt={name} width="800" height="600"
       loading="lazy" decoding="async" />
</picture>
```

All images use `loading="lazy"` to defer off-screen loads. Thumbnails use `loading="eager"`
for the above-fold first 6 results only.

## Build Integration

### Makefile Target

Add to root `Makefile`:

```makefile
.PHONY: renders renders-draft renders-missing

renders:          ## Batch-render all machines (production quality, ~2h)
	python3 tools/render_gallery.py --all --quality final

renders-draft:    ## Batch-render all machines (draft quality, ~10min)
	python3 tools/render_gallery.py --all --quality draft

renders-missing:  ## Print machine IDs without .blend scene files
	python3 tools/render_gallery.py --missing

renders-copy:     ## Copy render outputs to frontend/static/machines/
	python3 tools/render_gallery.py --copy-only
```

### CI Notes

Renders are NOT run in CI (2h wall time, GPU dependency). Instead:
- Rendered assets are committed to `frontend/static/machines/` using Git LFS.
- A CI job `renders-audit` runs `python3 tools/render_gallery.py --missing` and
  warns (does not fail) if any machine lacks a render file.
- The Git LFS pointer check ensures all `.webp` files are tracked correctly.

Add to `.gitattributes`:
```
frontend/static/machines/*.webp filter=lfs diff=lfs merge=lfs -text
frontend/static/machines/*.svg  -filter -diff  # SVGs are text, not LFS
```

## Checklist: Adding a New Machine Render

1. Add machine to `machine_registry.py` (already required for API).
2. Create `assets/renders/blender/machines/<id>/` directory.
3. Model the machine in Blender; save as `<id>.blend`.
4. Link material library files from `_template/materials/` for each part.
5. Position the camera using `_template/scene_template.blend` camera rig.
6. Write `<id>.render.json` manifest (copy from nearest similar machine).
7. Run `python3 tools/render_gallery.py --machine <id> --quality draft` and inspect.
8. Run `python3 tools/render_gallery.py --machine <id> --quality final`.
9. Copy outputs to `frontend/static/machines/` and commit with Git LFS.
10. Run `make renders-missing` to confirm zero missing entries.

## Modeling Fidelity Standards

All models must meet these minimum standards before a render is accepted:

| Standard         | Requirement                                                        |
|------------------|--------------------------------------------------------------------|
| Part count       | At least the primary mechanism (not just a box with a texture)     |
| Materials        | All visible parts use a MaterialLibrary-mapped Blender material     |
| Scale            | 1 Blender unit = 1 metre; machines modeled at real historical scale |
| Polygon budget   | <= 500k tris per scene (enforced by `_blender_render_worker.py`)    |
| UV unwrap        | Required for ivory, boxwood, organic_fiber (texture materials)      |
| Pose             | Machine in operational position (crank at rest, card loaded, etc.)  |
| Historical ref.  | At least one cited image/drawing used for proportions               |

Sources for reference drawings:
- Antikythera: Freeth et al. (2021) Nature, supplementary figures.
- Analytical Engine: Science Museum Group, London (object inventory 1862-83).
- Enigma: Crypto Museum technical drawings (enigmamuseum.com).
- ENIAC: Computer History Museum archive, accession 102646613.
- All others: See `docs/history/` and `docs/sources/SOURCE_MAP.md`.
